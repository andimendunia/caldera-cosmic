1. Infrastructure & System Requirements

1.1 Laragon Environment

Caldera runs on Laragon, a portable development environment for Windows that provides a complete web development stack.

Laragon Components (must be started):

  • Apache Web Server - Hosts the Laravel application
  • MySQL Database Server - Stores all application data
  • PHP 8.2+ - Runtime environment for Laravel
Important: Both Apache and MySQL are integrated components within Laragon and must be started through the Laragon control panel before running Caldera.

1.2 Caldera Command Manager (CCM)

The Caldera Command Manager is a Python-based GUI application that manages and monitors the essential background services required for Caldera's operation.

CCM Features:

  • Automated service management and monitoring
  • Real-time log viewing and analysis
  • Service status monitoring with automatic restart capabilities
  • Configuration management through GUI interface

CCM Location:

py/command_manager/dist/ccm_v1.0/ccm.exe

CCM Management Options:

  • Standalone Application: Run the CCM executable directly
  • Web Interface: Access through Caldera interface at Account Menu > Administration > Manage Daemons (superuser only)

1.3 Core Services

CCM manages these critical background services that must run continuously:

Queue Worker

php artisan queue:work

Processes background jobs and queued tasks essential for data processing and notifications.

Schedule Worker

php artisan schedule:work

Executes scheduled tasks including data cleanup, calculations, and automated routines.

CLM Polling (InsClmPoll)

php artisan app:ins-clm-poll --d

Climate monitoring polling daemon that collects environmental data from production areas.

CTC Polling (InsCtcPoll)

php artisan app:ins-ctc-poll --d

Calendar thickness control polling daemon for rubber calendering process monitoring.

STC Routine (InsStcRoutine)

php artisan app:ins-stc-routine --d

Stabilization temperature control routine daemon for environmental monitoring in stabilization chambers.

1.4 Scheduled Tasks

The Laravel scheduler runs automated tasks at specified intervals. These are defined in routes/console.php:

OMV Cleanup (Daily)

php artisan app:ins-omv-cleanup

Cleans up old OMV (Open Mill Validation) data and temporary files to maintain database performance.

Inventory Empty Resolution (Daily)

php artisan app:inv-empty-resolved

Resolves empty inventory items and processes pending stock adjustments.

User Preferences Sync (Every 5 Minutes)

php artisan app:sync-user-prefs

Synchronizes user preferences across sessions and maintains user setting consistency.

2. Project Structure & Architecture

2.1 Framework & Stack

  • Laravel 12 with PHP 8.2+ - Core framework
  • Livewire Volt - Reactive UI components (no traditional controllers)
  • MySQL - Primary database with extensive migrations
  • Vite - Asset bundling with TailwindCSS
  • Chart.js - Data visualization
  • Lucide - Icon library for consistent UI iconography
  • Modbus TCP client - Industrial equipment integration

2.2 Core Application Modules

Insights System (Manufacturing Monitoring)

OMV (Open Mill Validation)

Rubber mixing process monitoring with recipe management, real-time amperage tracking, and energy calculations.

CTC (Calendar Thickness Control)

Thickness monitoring for rubber calendering with machine performance analytics.

STC (Stabilization Temperature Control)

Environmental monitoring for temperature and humidity in stabilization chambers.

RDC (Rheometer Data Collection)

Automated rheometer testing with TC10/TC90 analysis for rubber quality control.

LDC (Leather Data Collection)

Hide/leather quality inspection system with worker performance tracking.

CLM (Climate Monitoring)

Environmental monitoring across production areas for optimal manufacturing conditions.

Inventory Management System

  • Multi-unit stock management with automatic conversions
  • Area-based authorization system
  • Circulation workflow (deposit/withdrawal) with approval processes
  • Multi-currency support with automatic calculations
  • Advanced analytics including stock aging and predictive reordering

Notification System

Built-in notification system for feature announcements and user communication:

  • Database-driven notifications stored in the notifications table
  • Feature announcements via FeatureAnnounce artisan command
  • User targeting options: superuser, active users, recent users, inventory users
  • Rich content support with icons (using Lucide icon library), content text, and clickable URLs
  • Announcement posts for detailed feature explanations with images

Sending Notifications

php artisan app:feature-announce

Interactive command that allows you to:

  • Select from predefined feature announcements
  • Choose target user groups (superuser, active users, recent users, etc.)
  • Preview notification content before sending
  • Automatically create announcement posts for detailed content

2.3 Database Architecture

  • Migration-driven development with extensive migration files
  • Prefixed table naming: ins_* (insights), inv_* (inventory)
  • Timestamps on most entities for audit tracking
  • JSON columns for flexible data storage (compositions, preferences)
  • Authorization models (*Auth) for granular permission control

2.4 Key Directory Structure

app/ ├── Console/Commands/ # Artisan commands for data processing ├── Models/ # Eloquent models for database entities ├── Livewire/ # Livewire Volt components (UI logic) ├── Http/Controllers/ # Minimal controllers (mostly API) ├── Policies/ # Authorization policies └── Services/ # Business logic services resources/views/livewire/ # Livewire Volt component views ├── insights/ # Manufacturing monitoring interfaces ├── inventory/ # Inventory management interfaces └── layout/ # Shared layout components py/ # Python integration scripts ├── command_manager/ # CCM application ├── omv-worker/ # OMV data processing └── ldc-worker/ # LDC image processing database/migrations/ # Database structure definitions

3. Server Deployment & Code Changes

3.1 Current Deployment Workflow

Since the Caldera server is not connected to the internet, code changes must go through an intermediate PC. The workflow involves three main steps:

  • Step A: Local Development to GitHub - Develop and push changes from your PC to the central repository
  • Step B: GitHub to Pak Khafit PC - Pull latest changes to the shared folder (172.70.66.102/mm/TIM DGT/_Repos/caldera-cosmic)
  • Step C: Pak Khafit PC to Caldera Server - Deploy changes and update the production environment

3.2 Deployment Commands

Step A Commands (Local to GitHub):

git add . git commit -m "Your commit message" git push origin main

Step B Commands (GitHub to Pak Khafit PC):

git pull origin main

Step C Commands (Pak Khafit PC to Caldera Server):

git pull mm main php artisan config:cache php artisan route:cache npm run build php artisan migrate:status
Important: Always check php artisan migrate:status to ensure all database migrations have been applied successfully. Run php artisan migrate if there are pending migrations.

3.3 Future Simplified Workflow

If the Caldera server gains internet connectivity in the future, the deployment process can be simplified to a direct two-step workflow:

Future Step 1: Local to GitHub

git add . git commit -m "Your commit message" git push origin main

Future Step 2: GitHub to Caldera Server (Direct)

git pull origin main php artisan config:cache php artisan route:cache npm run build php artisan migrate:status
Future Consideration: Direct internet connectivity will eliminate the need for the intermediate PC (Pak Khafit PC) and allow for more streamlined deployment processes, potentially including automated CI/CD pipelines.
